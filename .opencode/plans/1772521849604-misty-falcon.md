# Ployz Machine Add / Join / Invite System Plan

## Locked Constraints (from this discussion)
- Support many stored networks, but exactly **one active running network per daemon**.
- Daemon must **not auto-boot** any network on restart.
- Mesh command semantics stay strict:
  - `mesh create <name>` = create only
  - `mesh init <name>` = create + start
  - `mesh up <name>` = start existing
  - `mesh down` = stop runtime, keep config/data
  - `mesh destroy <name>` = destroy target network config/runtime
- No network override/reuse: existing network must be destroyed first.
- `machine` operations are valid only when connected daemon has a running network.
- Exception: `machine init <ssh-target> --network <name>` is a founder bootstrap command and is allowed without an existing running network.
- `machine add root@host` is orchestrated by the daemon; CLI chooses which daemon to talk to.
- `machine init root@host --network <name>` must bootstrap remote host and create/start founder network on that remote daemon.
- Join/bootstrap must handle founder NAT/private + new node public topology.
- Invite security model: **hybrid signed + stored (one-time, revocable)**.
- Eventually, destroy must be blocked until remote machines are decommissioned.
- Bootstrapping must eventually cover install/upgrade + Docker ensure/install + image store prerequisites across supported platforms.

## Target UX and API Shape

### CLI Surface
- Keep existing mesh commands (`create/init/up/down/destroy`) and add:
  - `mesh list`
  - `mesh status [name]`
  - `mesh join --token <token>` (bootstrap command; not a `machine` command)
- Add machine commands (running network required except `machine init` founder bootstrap):
  - `machine init <ssh-target> --network <name>`
  - `machine add <ssh-target>`
  - `machine list`
  - `machine status [machine-id]`
  - `machine remove <machine-id>`
  - `machine invite create [--ttl --uses]`

### Daemon RPC
- Extend `DaemonRequest` with structured machine and invite requests.
- Replace `DaemonResponse { ok, message }` with structured response:
  - `ok`, `code`, `message`, optional typed payload.
- Keep local socket protocol, add request variants for:
  - mesh list/status/join
  - machine init/add/list/status/remove
  - invite create/redeem

### Context/Target Model
- CLI is responsible for connecting to the correct daemon context (local or SSH-targeted daemon).
- Daemon executes join/provision orchestration from that context.
- `machine init` is the founder bootstrap path for a fresh remote host; all other machine commands require running network context.
- After successful `machine init`, CLI auto-adds/updates context for that daemon and switches to it.

## Core Domain & State Model

### Network Identity and Scoping
- Introduce immutable `NetworkId` (UUID/ULID) in network config.
- Keep `NetworkName` as display/operator key; all membership storage and subscriptions use `NetworkId`.
- Add persistent network catalog helpers (list by name/id, active runtime pointer).

### Runtime State Machine
- `Absent -> Created` via `mesh create`
- `Absent -> Running` via `mesh init`
- `Created -> Running` via `mesh up`
- `Running -> Created` via `mesh down`
- `Created|Running -> Absent` via `mesh destroy` (subject to distributed preconditions in later phase)
- Guard: any start op fails if another network is already running.

### Machine Join State Machine
- `InviteIssued -> InviteDelivered -> JoinRequested -> JoinValidated -> MembershipCommitted -> BootstrapPending -> Active`
- Failure lanes: `InviteExpired`, `InviteRedeemed`, `InvalidSignature`, `NetworkConflict`, `BootstrapUnreachable`.

## NAT/Public Bootstrap Strategy (explicit)
- Store endpoint candidates per machine: advertised/manual/observed/last_good.
- For founder NAT + joiner public:
  - founder initiates first authenticated WG traffic to joiner public endpoint.
  - joiner records founder observed endpoint from handshake source and promotes it.
- Keepalive defaults for NAT survival (`25s` equivalent behavior).
- Endpoint rotation uses health + backoff, but does not discard last known good endpoint eagerly.
- Explicit failure classification when direction is unreachable (`NAT_UNREACHABLE_DIRECTION`).

## Corrosion Client Strategy (updated)
- Decision: import `corro-client` from crates.io first (not git), pinned to an exact version.
- Add pinned dependency in root `Cargo.toml` (initially `corro-client = "=0.1.0-alpha.1"`), and wrap it behind `src/adapters/corrosion.rs`.
- Keep ployz-specific domain logic out of upstream dependency (no lifecycle policy/orchestration in dependency layer).
- Maintain a thin local adapter boundary so we can later switch to fork/internal crate without touching domain/control layers.
- Because this is an alpha crate with limited docs, lock behavior with adapter-level tests before rolling it into machine onboarding paths.
- If crates.io release has blockers, fallback is pinned git or pinned fork with identical adapter surface.

## Legacy Install Script Findings (from `../`)
- `../internal/remote/install.sh` is root-oriented and tightly couples release install + Docker install + systemd assumptions.
- `../scripts/bootstrap-remote.sh` is distro-branch-heavy shell logic with broad package manager branching and imperative host mutation.
- Keep these as historical operational references only; do not copy this shell-heavy model into new Rust control-plane logic.
- New system should keep provisioning steps explicit/idempotent in daemon onboarding state machine, with narrow shell use only where unavoidable.
- Target end state: daemon-managed bootstrap profiles that can ensure/install Docker, required networking tools, and Docker image-store prerequisites per platform, while remaining testable and resumable.

## Security Design (Chosen)
- Use hybrid invite token:
  - signed payload: `invite_id`, `network_id`, `issued_by`, `exp`, `jti`, optional bootstrap hints.
  - stored invite record in backend: one-time use counter, revocation, audit timestamps.
- Validation requires both valid signature and live backend redeemability.
- Prevent replay with one-time redemption + jti tracking.

## What We Reuse vs Reject from Go Version
- Reuse:
  - SSH `dial-stdio` style transport concept for remote daemon access.
  - keepalive and endpoint rotation ideas.
  - detach vs destroy lifecycle split.
- Reject:
  - shell-heavy brittle install flows as core logic.
  - unstructured/partial API behavior and implicit state transitions.

### Scenario tests (end-to-end)
- NAT founder + public joiner: successful add/join with observed endpoint promotion.
- Public founder + NAT joiner: successful reverse direction.
- Unreachable direction: deterministic `NAT_UNREACHABLE_DIRECTION` with remediation guidance.
- Destroy blocked until remote machines removed, then succeeds.
